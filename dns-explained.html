A beginner's field guide

DNS is the internet's post office.

You write a name on the envelope. Somewhere behind the scenes, a chain of sorting offices figures out the exact address and gets your letter delivered. This guide walks through that whole journey — interactively.

Why it exists

Names are for people. Numbers are for machines.

You don't memorize your friend's phone number — you save their name. Your phone does the lookup. The internet works exactly the same way, just with servers instead of a contacts app.

The phonebook entries

Every domain keeps a stack of record cards

Tap a card to flip it. Each record type answers a different question about a domain — an address, an alias, a mail route, or just a note.

Where the cards live

The zone file

All of a domain's records sit together in one plain-text file called a zone file — think of it as the domain's full index card, kept at the authoritative server.

example.com. IN A 192.0.2.10
www IN CNAME example.com.
mail IN A 192.0.2.20
example.com. IN MX 10 mail.example.com.
example.com. IN TXT "v=spf1 include:_spf.example.com ~all"

Each line is one record: a name, a type, and a value — exactly like the flip cards above.

Keeping the envelope sealed

DoH vs. DoT

Plain DNS queries travel like a postcard — anyone along the way can read them. These two approaches put the postcard in a sealed envelope instead.

DoH — DNS over HTTPS

Your DNS request rides inside normal encrypted web traffic, so it blends in with everything else your browser sends.

Port 443 · looks like regular HTTPS traffic

DoT — DNS over TLS

Your DNS request gets its own dedicated encrypted channel, separate from web traffic — easier for a network admin to identify and manage.

Port 853 · dedicated DNS channel

Quick aside: Twingate isn't a DNS server

It's a Zero Trust access tool. Instead of handing an employee the keys to the whole office (like a VPN), it checks who they are and unlocks only the one room — er, application — they're allowed into. Different job entirely, easy to mix up because it also sits "in the middle" of a connection.

Recap

The full route, one more time

You → cache → resolver → root → TLD → authoritative server → your browser, connected. Once this sticks, the next concepts fall into place fast: caching and TTL, split-horizon DNS, load balancing, and managed services like Route 53, Azure DNS, or Google Cloud DNS.
